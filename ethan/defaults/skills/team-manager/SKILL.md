# 团队管理 (Team Manager)

团队人员管理效率工具，覆盖绩效管理和任务委派两大场景。

> 时间线和定时任务管理已拆分到 `schedule-manager` 技能。本技能专注于**人**的管理。

## ⚡ 快速路径（优先匹配，命中即执行，勿绕路）

### 「整理进 people」

当用户说「把 xxx 整理进 people」「记到 people 里」时，**不要搜索、不要读参考文档**，直接执行：

1. 从用户消息或上下文中提取 `{姓名}: {事件}` 列表
2. 对每个人，写入 `~/.ethan/work/<业务>/people/{姓名}.md`，其中“业务”如“coze”，内容格式：
   ```
   # {YYYY-MM}

   ## {MM-DD}
   - {事件描述}（{HH:MM}）
   ```
   - 文件已存在：在当月标题下追加当日条目（如当日已有则追加在最后一条下面）
   - 文件不存在：创建文件，写入上述格式
3. 回复确认：列出写入了哪些人、各写了什么

**总计最多 3 步**：提取 → 写入 → 确认。

### 「业务进展 / 工作进展」

当用户说「更新业务进展」「更新工作进展」或发送文档/链接要求更新进展时，**不要 web_search、不要 knowledge_search**，直接执行：

**判断标准**：只要用户意图是"把某些人做的事/项目进展记录下来"，即使没有精确命中 trigger 关键词，也应走此路径。

1. 读取文档/上下文内容，提取**每个人做的最关键的事**（精简，不要太多太细）
2. **写 people**：按人写入 `~/.ethan/work/<业务>/people/{姓名}.md`，格式同上
3. **写 project**：先确定所属业务目录（`ls ~/.ethan/work/` 查看已有业务目录），再写入 `~/.ethan/work/{业务名}/project/{项目名}.md`，格式：
   ```markdown
   # {事项/方向名称}
   > {路线或 checkpoint 描述，含时间节点}

   - {进展条目1}
   - {进展条目2}

   ---
   ```
   - ⚠️ **绝不写到 `~/.ethan/work/project/`**，必须写到 `~/.ethan/work/{业务名}/project/`
   - 业务名根据已有目录和内容归属判断
   - 项目名由模型根据内容判断，无法归类时用 `default.md`
   - 已完结事项标题前加 `[DONE]`
   - 进展涉及具体人时链接到 people：`见 [people/小王.md](../../people/小王.md#07-21)`
4. 回复确认：列出更新了哪些人、哪些项目

**总计最多 4 步**：读取 → 写 people → 写 project → 确认。

### 「文档/链接收藏」

当用户发送文档链接、或说「收藏这个」「存一下」，或 Agent 在处理过程中**解析了任何文档/链接**时，**自动追加收藏记录**：

1. 解析文档：提取标题、链接、一句话摘要、作者（可选）、日期
2. 判断归属业务：`ls ~/.ethan/work/` 查看已有业务目录，根据内容判断归属，不确定时询问用户
3. 追加到 `~/.ethan/work/{业务名}/docs/{分类}.md`（⚠️ 绝不写到 `~/.ethan/work/docs/`），格式：
   ```markdown
   ## {文档标题}
   - 链接：{url}
   - 一句话介绍：{摘要}
   - 作者：{作者}
   - 日期：{YYYY-MM-DD}
   ```
4. 回复中附带「已收藏到 {业务名}/docs/{分类}.md」

**注意**：这是**附带动作**——即使用户的主要意图是"更新进展"或"文档分析"，只要解析了链接就顺带收藏。

## 🎯 两大核心场景

### 场景 A：绩效管理

持续收集团队成员工作事件，绩效季自动汇总并生成评估建议。

**能力**：
1. **手动记录** — 你随时说，Agent 自动写入对应成员的日志
2. **文档分析** — 转发下属文档时，提取亮点数据、架构思路、业务价值
3. **CR 统计** — 按人汇总代码产出、质量问题、需求归属
4. **绩效汇总** — 绩效季按人输出全景 + 初步评估建议

详见：
- `references/people-log.md` — 人员日志格式与写入规则
- `references/analysis.md` — 文档分析 & CR 分析模板
- `references/review-guide.md` — 绩效评估报告生成指南

### 场景 B：任务委派

将任务分配给团队成员，自动拆解阶段、创建飞书任务、设置 checkpoint 提醒。

**能力**：
1. **任务拆解** — 接收描述，拆分为可跟踪的阶段性子任务
2. **创建飞书任务** — 为每个阶段创建飞书任务，设置到期日和提醒
3. **Checkpoint 跟踪** — 关键节点前自动提醒，确保按计划推进
4. **绩效联动** — 任务完成/延期自动写入对应人员的日志

详见：
- `references/workflow.md` — 委派流程 SOP 与 Checkpoint 拆分指南

## 📁 数据存储

### 目录结构

数据存储在 `~/.ethan/work/` 下（work scene 由 `schedule-manager` 负责创建和维护）：

```
~/.ethan/work/
├── team.yaml            # 团队配置（成员、项目、CR 平台映射）
├── people/              # 按人按月按日记录（跨业务，唯一记录源）
│   ├── {姓名}.md
│   └── _self.md         # 管理者自己的日志
├── {业务名}/            # 每个独立业务一个一级目录（如产品线/业务线）
│   ├── scope.md         # 该业务当前负责范围与历史范围
│   ├── project/         # 该业务下的项目进展（链接到 people）
│   │   ├── {项目名}.md
│   │   └── default.md   # 无法归类的项目
│   ├── docs/            # 该业务下的文档/链接收藏
│   │   ├── {分类}.md
│   │   └── default.md   # 无法归类的收藏
│   └── cr-reports/      # 该业务下的 CR 分析报告
│       └── {YYYY}-W{NN}.md
├── reviews/             # 绩效综合评估（跨业务，顶层）
│   └── {YYYY}-Q{N}-draft.md
└── .group_scan_state.json
```

**层级关系**：`业务` → `业务方向` → `项目` → `需求`

- **业务**：`work/` 下的一级目录，代表一条产品线/业务线
- **业务方向**：业务内的大分类（如"桌面端"、"Web 端"），记录在 `scope.md` 中，用于组织项目归属
- **项目**：`project/` 下的 md 文件，由用户告知或 Agent 根据内容整理（避免重叠）
- **需求**：项目文件内的各条进展/事项

**目录设计原则**：

1. **一级目录 = 独立业务**：`work/` 下除 `people/`、`reviews/`、`team.yaml` 外，每个一级文件夹代表一条独立业务线。业务文件夹由模型根据内容自动判断归属，新业务自动创建目录
2. **`people/` 跨业务**：人员日志不按业务拆分，因为同一个人可能参与多条业务线
3. **`reviews/` 跨业务**：绩效评估是对人的综合评估，不属于单一业务
4. **业务目录内按资料类型组织**：`scope.md`（范围定义）、`project/`（项目进展）、`docs/`（文档收藏）、`cr-reports/`（代码审查报告）等
5. **业务目录可增减**：负责的业务范围可能变化，目录随业务自然增减，无需预定义

> ⚠️ **严禁在 `work/` 根目录下创建 `scope.md`、`project/`、`docs/`、`cr-reports/` 等文件或目录。** 这些资料**必须且只能**存放在对应的业务子目录内（如 `work/{业务名}/scope.md`）。`work/` 根目录下只允许存在：`team.yaml`、`people/`、`reviews/`、`.group_scan_state.json`、以及各业务文件夹本身。

**写入前必检**：任何写入 project/docs/scope/cr-reports 的操作，Agent 必须先：
1. 确定内容归属哪个业务（读已有业务目录 or 从上下文判断）
2. 确认目标路径是 `~/.ethan/work/{业务名}/...` 而非 `~/.ethan/work/...`
3. 若业务目录不存在，先创建目录和 `scope.md`

> `timelines.yaml` 和 `.timeline_state.json` 由 `schedule-manager` 管理，不在本技能范围内。

### people/ 日志格式

按人按月按日记录，文件命名 `{姓名}.md`，最新月份在上，月份间用 `---` 分隔：

```markdown
# 2026-07

## 07-21
- [亮点] 完成了支付模块联调，性能提升 30%（14:30）
- [问题] 线上 P1 告警，Redis 连接池耗尽（03:15）
- 参加了周会（10:00）

## 07-20
- 提出重构库存查询方案，被采纳（10:00）

---

# 2026-06

## 06-28
- 修复线上 P1 告警，根因是 Redis 连接池耗尽（03:15）
```

**标签约定**：`[亮点]` / `[问题]` / `[进展]` 可选，普通事件不加标签。

详见 `references/people-log.md`。

### project/ 项目进展格式

按项目组织进展视图，存放于 `~/.ethan/work/{业务名}/project/{项目名}.md`（名字由模型自行判断，无对应项目时用 `default.md`）。
已完结的项目/事项在标题前加 `[DONE]`。格式：

```markdown
# 桌面端功能
> 7.25 前先实现基础功能开发，8.15 实现 xxx，8.24 完成 xxx

- xxx 开发中
- xxx 已提测
- xxx 已完成上线

---

# [DONE] 桌面端 BUA
> 6.30 完成灰度上线

- 完成了 xxx
```

**规则**：
- 每个 `#` 标题是一个独立事项/方向
- `>` 引用块写路线或 checkpoint 描述（如有时间节点，可让 `schedule-manager` 创建提醒）
- 列表项是当前状态/最新进展
- 标题间用 `---` 分隔
- 进展条目中涉及具体人时，链接到 people：`见 [people/小王.md](../../people/小王.md#07-21)`

### docs/ 文档收藏格式

收藏解析过的文档/链接，存放于 `~/.ethan/work/{业务名}/docs/` 下。文件按分类命名（如 `技术方案.md`、`竞品分析.md`），无法进一步分类时用 `default.md`。

```markdown
## 支付模块重构方案
- 链接：https://xxx.feishu.cn/docx/xxx
- 一句话介绍：描述了支付模块从单体到微服务的拆分方案，含数据迁移策略
- 作者：小王
- 日期：2026-07-21
```

**规则**：
- 每解析一个文档/链接就追加一条
- `作者` 字段可选，知道就写
- `日期` 是文档创建或最后修改日期（从文档元信息中获取，获取不到则用当天）
- 最新条目追加在文件末尾

### scope.md 业务范围格式

每个业务目录下必须有一个 `scope.md`，记录当前负责的业务范围及历史变更。格式：

```markdown
# current

- {业务方向1}
  - {项目A}: {简介}
    - 关联工作：project/{项目A}.md
  - {项目B}: {简介}
    - 关联工作：project/{项目B}.md
- {业务方向2}
  - {项目C}: {简介}
    - 关联工作：project/{项目C}.md

# {起止时间段}（如 2026.01 ~ 2026.05）

- {已不再负责的业务方向}
  - {项目X}: {简介}
```

**规则**：
- `# current` 下列出当前负责的范围，`# {时间段}` 下列出历史负责过但已移交/结束的范围
- 一级列表项 = **业务方向**（大分类，如"桌面端"、"Web 端"），控制数量，不要太细
- 二级列表项 = **项目**，带简介和关联工作链接
- 业务方向由 Agent 在添加项目/需求时自动抽取归类，保持精简（通常 3-5 个为宜）
- 项目通常由用户告知，Agent 也可根据内容自行整理，但需避免重叠
- 当业务范围变化（移交/新增），将旧条目移到对应时间段标题下
- 新建业务目录时，Agent 应同时创建 `scope.md` 并写入初始范围

## 🔑 触发识别规则

### 绩效记录

| 用户输入模式 | Agent 行为 |
|---|---|
| `记一下，{姓名}...` | 识别人名 → 写入 people/{姓名}.md（按月去重） |
| `我的记录：...` | 写入 people/_self.md |
| `{姓名}做了/完成/修复了...` | 识别人名和事件 → 写入 people/{姓名}.md |
| `整理进 people` / `记到 people` | **快速路径**（见上方） |
| `更新业务进展` / `更新工作进展` | **快速路径：业务进展**（见上方） |
| `{姓名}最近怎么样` | 读取 people/{姓名}.md 最近 1-2 个月记录，综合回顾 |
| `{姓名}的事` / `{姓名}做了什么` | 读取 people/{姓名}.md，按时间顺序总结 |

**写入规则**：
- 单一数据源：所有事件只写入 `people/{姓名}.md`
- 标签可选：`[亮点]` `[问题]` `[进展]` 用于标记事件类型，普通事件不加标签
- 按月去重：写入前读取当月已有记录，完全相同文本静默跳过，核心动词+对象重叠则询问合并/替换/另记
- `[进展]` 标签明确表示是事件更新而非重复，写入时不触发去重询问

### 文档分析

当用户发送文档链接或说「帮我看看这个文档」时：
1. 读取文档内容
2. 提取亮点（数据/架构/业务价值/风险）
3. 询问归属人
4. 写入 people/{姓名}.md，带 `[亮点]` 或 `[问题]` 标签

### CR 分析

用户说「CR 汇总」「代码产出统计」「本周 CR」时，或定时任务触发周度 CR 汇总时：
1. 读取 `scope.md` 确定当前业务方向和关注的仓库/项目
2. 调用 Codebase API 拉取对应仓库最近一周的 MR 数据（按团队成员过滤）
3. 按人统计：MR 数量、代码行数、涉及仓库、关联需求
4. 识别质量问题（P0/P1）和正面信号（见 `references/analysis.md`）
5. 输出周报到 `~/.ethan/work/{业务名}/cr-reports/{YYYY}-W{NN}.md`
6. 提取值得记录的事件写入 `people/{姓名}.md`（带 `[亮点]` 或 `[问题]` 标签）
7. 回复摘要（关键产出 + 质量趋势）

**定时任务建议**（用户可配置到 `schedule-manager`）：

> 每周五下午对本周团队 MR 做一次 CR 汇总。读取 team.yaml 获取成员和关注仓库列表，调用 Codebase API 拉取本周 MR，按 team-manager 技能的 CR 分析流程生成周报，写入对应业务目录的 cr-reports/，并将关键事件同步到 people/。完成后发送摘要给我。

### 任务委派

用户说「安排一下」「交给 XX」时：
1. 解析任务内容、被委派人
2. 确认 DDL
3. 拆分 checkpoint
4. 创建飞书任务 + 设置提醒（提醒创建交给 `schedule-manager`）

### 群消息提取

用户说「帮我监控这个群」时：
1. 获取群 chat_id（若已在群内 @bot 则自动取当前群）
2. 询问扫描频率（默认每天 22:00）和时间窗口（默认最近 24 小时）
3. 创建一个定时任务（通过 `schedule-manager` 的 `schedule_create`），prompt 使用群扫描模板
4. 定时任务触发时，Agent 按模板步骤拉取消息、识别候选事件、输出去重清单
5. 用户在原渠道回复「入库 1,3,5」确认后写入对应成员的 people/{姓名}.md

详见 `references/group-scan-prompt.md`。

## ⚠️ 约束

- **确认后再创建**：任务拆分方案、飞书任务创建，必须经用户确认
- **不做最终绩效判定**：Agent 只提供汇总和建议，打分由管理者决定
- **隐私边界**：所有记录仅存本地，不上传外部服务
- **客观记录**：people 日志以事实描述为主，避免主观判断词
- **不越权**：Agent 不自行修改 DDL 或取消任务

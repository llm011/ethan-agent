# 团队管理 (Team Manager)

一站式管理者效率工具，覆盖三大场景。

## 🎯 三大核心场景

### 场景 A：绩效管理

持续收集团队成员工作事件，绩效季自动汇总并生成评估建议。

**能力**：
1. **手动记录** — 你随时说，Agent 自动写入对应成员的日志
2. **文档分析** — 转发下属文档时，提取亮点数据、架构思路、业务价值
3. **CR 统计** — 按人汇总代码产出、质量问题、需求归属
4. **绩效汇总** — 绩效季按人输出全景 + 初步评估建议

详见：
- `references/people-log.md` — 人员日志格式与写入规则
- `references/analysis.md` — 文档分析 & CR 分析模板
- `references/review-guide.md` — 绩效评估报告生成指南

### 场景 B：任务委派

将任务分配给团队成员，自动拆解阶段、创建飞书任务、设置 checkpoint 提醒。

**能力**：
1. **任务拆解** — 接收描述，拆分为可跟踪的阶段性子任务
2. **创建飞书任务** — 为每个阶段创建飞书任务，设置到期日和提醒
3. **Checkpoint 跟踪** — 关键节点前自动提醒，确保按计划推进
4. **场景标签** — 区分工作/生活/健康等场景，互不干扰
5. **绩效联动** — 任务完成/延期自动写入对应人员的日志

详见：
- `references/workflow.md` — 委派流程 SOP 与 Checkpoint 拆分指南
- `references/scenes.md` — 场景标签定义与自动推断规则

### 场景 C：时间线驱动

管理绩效周期、OKR、季度汇报等周期性事件，在关键节点自动触发提醒和操作。

**能力**：
1. **声明式时间线** — 用户配置 anchor + phases + actions
2. **自动展开调度** — Engine 将配置"编译"为具体的定时任务
3. **阶段感知** — 知道当前处于哪个时间线的哪个阶段
4. **周期轮转** — 本周期结束后自动准备下一周期的任务

详见：
- `references/timeline-engine.md` — 时间线解析引擎 SOP
- `templates/timelines.yaml.example` — 时间线配置模板

## 📁 数据存储

所有数据存储在 `~/.ethan/work/` 目录：

```
~/.ethan/work/
├── team.yaml                # 团队配置（成员、项目、CR 平台映射）
├── timelines.yaml           # 时间线配置（绩效周期、OKR 等）
├── .timeline_state.json     # 时间线运行状态（已触发动作记录）
├── people/
│   ├── {姓名}.md            # 按人按月按日记录（唯一记录源）
│   └── _self.md             # 管理者自己的日志
├── cr-reports/
│   └── {YYYY}-W{NN}.md     # 周度 CR 分析报告
├── reviews/
│   └── {YYYY}-Q{N}-draft.md # 绩效综合评估草稿
└── exports/                 # 导出文件存放目录
    └── timelines-{YYYY-MM-DD}.yaml
```

### people/ 日志格式

按人按月按日记录，文件命名 `{姓名}.md`，最新月份在上，月份间用 `---` 分隔：

```markdown
# 2026-07

## 07-21
- [亮点] 完成了支付模块联调，性能提升 30%（14:30）
- [问题] 线上 P1 告警，Redis 连接池耗尽（03:15）
- 参加了周会（10:00）

## 07-20
- 提出重构库存查询方案，被采纳（10:00）

---

# 2026-06

## 06-28
- 修复线上 P1 告警，根因是 Redis 连接池耗尽（03:15）
```

**标签约定**：`[亮点]` / `[问题]` / `[进展]` 可选，普通事件不加标签。

详见 `references/people-log.md`。

## 🔑 触发识别规则

### 绩效记录

| 用户输入模式 | Agent 行为 |
|---|---|
| `记一下，{姓名}...` | 识别人名 → 写入 people/{姓名}.md（按月去重） |
| `我的记录：...` | 写入 people/_self.md |
| `{姓名}做了/完成/修复了...` | 识别人名和事件 → 写入 people/{姓名}.md |
| `整理进 people` / `记到 people` | **快速路径**（见下方） |
| `{姓名}最近怎么样` | 读取 people/{姓名}.md 最近 1-2 个月记录，综合回顾 |
| `{姓名}的事` / `{姓名}做了什么` | 读取 people/{姓名}.md，按时间顺序总结 |

#### ⚡ 快速路径：「整理进 people」

当用户说「把 xxx 整理进 people」「记到 people 里」时，**不要搜索、不要读参考文档**，直接执行：

1. 从用户消息或上下文中提取 `{姓名}: {事件}` 列表
2. 对每个人，写入 `~/.ethan/work/people/{姓名}.md`，格式：
   ```
   # {YYYY-MM}

   ## {MM-DD}
   - {事件描述}（{HH:MM}）
   ```
   - 文件已存在：在当月标题下追加当日条目（如当日已有则追加在最后一条下面）
   - 文件不存在：创建文件，写入上述格式
3. 回复确认：列出写入了哪些人、各写了什么

**总计最多 3 步**：提取 → 写入 → 确认。无需额外搜索或读取参考文档。

**写入规则**：
- 单一数据源：所有事件只写入 `people/{姓名}.md`，不再有 cases/ 目录
- 标签可选：`[亮点]` `[问题]` `[进展]` 用于标记事件类型，普通事件不加标签
- 按月去重：写入前读取当月已有记录，完全相同文本静默跳过，核心动词+对象重叠则询问合并/替换/另记
- `[进展]` 标签明确表示是事件更新而非重复，写入时不触发去重询问
- 详细格式与写入流程见 `references/people-log.md`

### 文档分析

当用户发送文档链接或说「帮我看看这个文档」时：
1. 读取文档内容
2. 提取亮点（数据/架构/业务价值/风险）
3. 询问归属人
4. 写入 people/{姓名}.md，带 `[亮点]` 或 `[问题]` 标签

### CR 分析

用户说「CR 汇总」「代码产出统计」时：
1. 调用 Codebase API 拉取 MR 数据
2. 按人统计产出量、质量问题、需求归属
3. 输出报告 + 写入 case

### 任务委派

用户说「安排一下」「交给 XX」时：
1. 解析任务内容、被委派人
2. 确认 DDL
3. 拆分 checkpoint
4. 创建飞书任务 + 设置提醒

### 时间线管理

用户说「时间线」「绩效周期」「下一阶段是什么」时：
1. 读取 `timelines.yaml`
2. 展示当前各时间线状态
3. 可手动触发下一阶段动作

### 配置时间线

用户告知周期性事件信息时（如「每年 7 月份开始半年绩效」「1 月全年绩效」）：
1. 解析用户描述，提取：锚点日期、周期类型、阶段划分
2. 如果 `timelines.yaml` 不存在，从 template 创建
3. 生成时间线配置草稿，展示给用户确认
4. 确认后写入 `timelines.yaml`
5. 触发 Timeline Engine 的 sync_scheduler 同步到定时任务池

### 导出/导入时间线

用户说「导出时间线」「备份时间线」「导入时间线」「恢复时间线」时：
- **导出**：将 `timelines.yaml` + `.timeline_state.json` 打包为 YAML/JSON，写到 `exports/`
- **导入**：读取导出文件，校验格式，写入配置，重新同步 scheduler

详见 `references/timeline-engine.md` 中「导出与迁移」章节。

### 群消息提取

用户说「帮我监控这个群」时：
1. 获取群 chat_id（若已在群内 @bot 则自动取当前群）
2. 询问扫描频率（默认每天 22:00）和时间窗口（默认最近 24 小时）
3. 创建一个 `recurring` 类定时任务，prompt 使用群扫描模板填充群信息
4. 定时任务触发时，Agent 按模板步骤拉取消息、识别候选事件、输出去重清单
5. 用户在原渠道回复「入库 1,3,5」确认后写入对应成员的 people/{姓名}.md

详见 `references/group-scan-prompt.md` — 完整的扫描 prompt 模板、状态文件格式与使用方式。

本质上群监控 = 定时任务 + prompt 模板，不需要预配置。

## 🏷️ 定时任务分类

所有通过本技能产生的定时任务分三类（详见 `references/timeline-engine.md`）：

| 类别 | 标识 | 图标 | 说明 |
|---|---|---|---|
| 一次性 | `one_off` | ⚡ | 执行一次后删除（如：提醒小王明天交方案） |
| 周期性 | `recurring` | 🔄 | 固定频率重复（如：每周五发 CR 周报） |
| 时间线 | `timeline` | 📅 | 由时间线阶段驱动（如：绩效季前2周汇总提醒） |

界面展示时按类别分 Tab，时间线类任务额外支持按时间线分组。

## 🏷️ 场景标签

所有任务/提醒都带场景标签（`work` / `life` / `health` / `study` / `finance` / `social`），参见 `references/scenes.md`。

## ⚙️ 首次配置

1. 复制 `templates/team.yaml.example` → `~/.ethan/work/team.yaml`，填入团队信息
2. （可选）复制 `templates/timelines.yaml.example` → `~/.ethan/work/timelines.yaml`，配置时间线
3. 开始使用——群监控、文档分析、CR 统计均为用户主动触发，无需预配置

## ⚠️ 约束

- **确认后再创建**：任务拆分方案、飞书任务创建，必须经用户确认
- **不做最终绩效判定**：Agent 只提供汇总和建议，打分由管理者决定
- **隐私边界**：所有记录仅存本地，不上传外部服务
- **客观记录**：case 以事实描述为主，避免主观判断词
- **不越权**：Agent 不自行修改 DDL 或取消任务

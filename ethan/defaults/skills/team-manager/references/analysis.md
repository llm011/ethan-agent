# 文档与 Code Review 分析模板

## 文档分析

当用户转发文档（下属写的技术方案、周报、设计文档等）时，按以下框架提取信息。

### 提取维度

#### 1. 关键数据（量化成果）

提取文档中所有量化指标：
- 性能提升/下降百分比
- 成本变化
- 用户量/DAU/转化率等业务指标
- 时间节省
- 系统可用性指标（SLA、P99 延迟等）

**提取格式**: `[指标名] 从 [旧值] → [新值]（提升/降低 X%）`

#### 2. 技术方案亮点

- 核心架构选型及理由
- 创新性解法（非常规思路）
- 技术难点的突破方式
- 对系统稳定性/可维护性的提升设计
- 引入的新技术栈或工具及其价值

**关注信号**: 有清晰的 trade-off 分析、有对比方案、有降级策略

#### 3. 业务价值

- 解决了什么用户痛点
- 对业务指标的预期/实际影响
- 覆盖的用户场景和影响面
- 对上下游的正向影响

#### 4. 思路与方法论

- 问题拆解方式是否清晰
- 是否有系统性的思考框架
- 是否考虑了边界条件和异常场景
- 文档本身的表达质量（结构化、可读性）

#### 5. 风险与不足（如有）

- 方案中未充分考虑的场景
- 潜在的性能/安全/稳定性风险
- 缺失的关键环节（如监控、回滚、灰度策略）

### 输出格式

```markdown
## 文档分析：{文档标题}

**作者**: {姓名}
**日期**: {YYYY-MM-DD}
**类型**: 技术方案 / 周报 / 设计文档 / 复盘报告

### 亮点提取

1. [关键数据] ...
2. [技术方案] ...
3. [业务价值] ...

### 值得记录的 Case

- case 1: {描述} → 建议标签: [技术深度, 亮点]
- case 2: {描述} → 建议标签: [业务价值, 产出]

### 待关注点（如有）

- ...
```

## Code Review 分析

### 周度 CR 统计报告模板

```markdown
## CR 周报：{YYYY}-W{NN}

**统计周期**: {开始日期} ~ {结束日期}

### 产出概览

| 成员 | MR 数 | 新增行数 | 删除行数 | 涉及仓库 |
|---|---|---|---|---|
| {姓名} | N | +XXX | -YYY | repo1, repo2 |

### 需求归属

| 成员 | 完成需求 | 关联 MR |
|---|---|---|
| {姓名} | {需求描述} | MR#1234, MR#1235 |

### 质量问题

| 成员 | 严重级别 | 问题描述 | 是否重复 | MR |
|---|---|---|---|---|
| {姓名} | P0/P1 | {问题描述} | 是/否 | MR#1234 |

### 质量趋势（本季度累计）

| 成员 | P0 累计 | P1 累计 | 同类重复 | 趋势 |
|---|---|---|---|---|
| {姓名} | N | N | 有/无 | ↑改善 / →持平 / ↓恶化 |

### 值得记录的 Case

- {姓名}: {正面/负面描述} → 建议标签
```

### CR 质量问题分类

| 级别 | 定义 | 绩效影响 |
|---|---|---|
| **P0** | 线上事故级：数据丢失、安全漏洞、服务不可用 | 重要负面信号 |
| **P1** | 潜在风险：资源泄漏、性能退化、逻辑错误 | 需关注，偶发可接受 |
| **P2** | 规范类：命名不规范、缺少注释、风格问题 | 仅在总结中带过 |

### 重复问题判定

当同一人在同一季度内出现 **2 次以上** 同类 P0/P1 问题时，标记为「重复问题」：
- 同类定义：相同根因类型（如都是错误处理缺失、都是并发安全问题）
- 记录到 case 中并打标签 `[重复问题, 质量问题]`
- 绩效汇总时作为「技术短板」信号呈现

### 正面 CR 信号

以下情况值得记录为正面 case：
- 主动重构老代码，提升可维护性
- MR 中包含完善的单测/集成测试
- 主动添加监控/告警/降级策略
- Review 他人代码时给出高质量建议
- 大规模改动零质量问题

# 群消息扫描 Prompt 模板

本模板由「帮我监控这个群」生成的定时任务调用。每次触发时，Agent 拉取指定群最近一段时间的消息，按以下规则扫描并输出候选事件，最终由用户确认后写入对应成员的 people/ 日志。

## 任务上下文

- **场景**：团队管理 / 人员事件记录
- **数据源**：飞书群消息（通过 `lark-im` 的 `+chat-messages-list` 或 `+messages-search`）
- **输出目标**：候选事件列表，等待用户确认后写入 `~/.ethan/work/people/{姓名}.md`
- **日志格式**：参见 `references/people-log.md`

## 完整 Prompt

```
你是团队管理助手。下面是你的任务：扫描指定飞书群的消息，识别其中值得记录的人员事件。

【群信息】
- 群名称：{group_name}
- 群 chat_id：{chat_id}
- 扫描时间窗口：最近 {window_hours} 小时（{start_time} ~ {end_time}）
- 本次扫描的批次 ID：{batch_id}（用于消息级去重）

【团队配置】
成员清单（仅识别这些人，其他人忽略）：
{members_brief}

其中 members_brief 格式为：
- 姓名 | 角色 | 负责方向 | 关联项目

【步骤 1：拉取消息】
调用 lark-im 工具，按以下条件拉取：
- chat_id：{chat_id}
- 时间范围：start_time ~ end_time
- 排除：bot 自己发的消息、纯表情、纯「收到/好的/嗯」等无信息量回复
- 排除：本群中你已经处理过的消息（通过 state 文件的 processed_message_ids 去重）

【步骤 2：识别候选事件】
对每条候选消息，判断是否属于以下任一类别。**严格按信号判定，不要过度提取**：

A. 亮点信号（标记 [亮点]）
- 主动承担额外职责、跨团队协作
- 解决了线上问题/告警，并给出根因分析
- 提出有价值的方案/建议（被采纳或进入讨论）
- 交付物有量化成果（性能提升 X%、用户增长 Y、成本下降 Z）
- 公开分享、技术输出、文档产出

B. 质量信号（标记 [问题]）
- 出现 P0/P1 级别的代码/线上问题
- 同类问题重复出现（需要标注"重复"）
- 进度延期、阻塞未及时上报
- 方案有明显遗漏或风险

C. 协作信号（普通事件，不加标签）
- 主动协调资源、推动跨团队对齐
- 帮助他人解答问题、Review 给出高质量建议
- 主持/组织关键会议

D. 进展更新（标记 [进展]）
- 之前记录过的事件有了新进展（如"开始 X" → "X 完成"）
- 同一项目的阶段性节点

**忽略信号**（以下情况不提取）：
- 日常寒暄、表情互动
- 纯进度同步（"已完成"、"已合并"）除非有量化数据
- 无量化数据的工作汇报（除非有突出亮点）
- 来自非团队成员的消息
- 对方案/讨论的简单附和

【步骤 3：提取字段】
对每个候选事件，提取以下字段：

| 字段 | 提取规则 |
|---|---|
| `name` | 团队成员姓名（从消息发送者或内容中识别） |
| `date` | 消息发送日期（YYYY-MM-DD） |
| `time` | 消息发送时间（HH:MM，24 小时制） |
| `tag` | `[亮点]` / `[问题]` / `[进展]` 或空（普通事件） |
| `content` | 客观事实描述，量化优先，**避免主观评价词** |
| `project` | 若消息提到项目名则填，否则留空 |
| `ref` | 飞书消息链接 |
| `context` | 简要背景（如"在讨论 X 方案时"），可选 |

【步骤 4：按月去重】
- 按成员分组，读取该成员 `~/.ethan/work/people/{姓名}.md` 中**当月**已有记录
- 去重规则：
  - 完全相同文本 → 静默跳过
  - 核心动词+对象重叠（如"完成支付联调" vs "搞定了支付对接"）→ 询问用户：合并/替换/另记
  - 候选事件带 `[进展]` 标签 → 不触发去重询问（明确是事件更新）
- 跨月事件不做去重（概率很小，不需要）

【步骤 5：输出候选清单】
按以下格式输出，**不要直接写入 people/ 文件**，等待用户确认：

```
## 群扫描候选事件 — {batch_id}

扫描范围：{group_name} · {start_time} ~ {end_time}
本次共识别 {N} 条候选事件：

### 候选 1
- 成员：{姓名}
- 日期：{YYYY-MM-DD} {HH:MM}
- 标签：{tag 或 "无"}
- 内容：{客观事实描述}
- 项目：{project 或 "—"}
- 引用：{消息链接}

### 候选 2
...

【处理建议】
- 建议入库：候选 1, 3, 5
- 建议合并：候选 2 与当月已有记录"{已有事件摘要}"（核心事件相同）
- 建议丢弃：候选 4（信息量不足）

等待用户确认后，将"建议入库"的候选写入对应成员的 people/{姓名}.md。
```

【写入格式】
用户确认后，按 people-log.md 的格式追加到对应成员的 md 文件：

```markdown
## {MM-DD}
- {tag}{内容}（{HH:MM}）
```

若当天日期标题已存在，直接在该日期下追加新条目；否则先添加日期标题再追加。

【约束】
1. **不越权写入**：未经用户确认，绝不修改 people/ 目录下的任何文件
2. **客观描述**：事件内容只描述事实，不带主观评价（如"做得很好"）
3. **隐私边界**：识别到的事件仅存本地，不发送到其他群或个人
4. **低噪声**：宁可漏一两件，也不要每天推 10+ 条候选让用户疲劳
5. **批次去重**：每次扫描记录 batch_id 和 processed_message_ids 到 `~/.ethan/work/.group_scan_state.json`，下次扫描时跳过已处理的消息
```

## 使用方式

### 用户首次配置群监控

用户：「帮我监控这个群」

Agent 行为：
1. 询问要监控的群名称或 chat_id（若消息中已 @bot 在群内则直接取当前群）
2. 询问扫描频率（默认：每天 22:00 扫描当天消息）
3. 询问时间窗口（默认：最近 24 小时）
4. 读取 `~/.ethan/work/team.yaml` 获取团队成员清单
5. 调用 `ScheduleCreateTool` 创建定时任务：
   - `category: recurring`
   - `cron: "0 22 * * *"`（或用户指定频率）
   - `prompt`: 本模板填入群信息后的完整内容
   - `title: "[群扫描] {群名称}"`
6. 在 `~/.ethan/work/.group_scan_state.json` 中记录该群的扫描状态

### 定时任务触发时

调度器按时触发，将填好的 prompt 作为新对话发给 Agent。Agent 按模板步骤执行，最终把候选清单作为回复返回。

由于定时任务的回复会发回原渠道（飞书群/私聊），用户可直接在飞书中回复「入库 1,3,5」来确认。

### 状态文件格式

`~/.ethan/work/.group_scan_state.json`:

```json
{
  "groups": {
    "{chat_id}": {
      "group_name": "团队反馈群",
      "last_scan_at": "2026-07-21T22:00:05",
      "last_batch_id": "batch_20260721_220000",
      "processed_message_ids": ["om_xxx1", "om_xxx2"],
      "cron": "0 22 * * *",
      "window_hours": 24
    }
  }
}
```

### 暂停/恢复/删除监控

| 用户指令 | Agent 行为 |
|---|---|
| 「暂停监控这个群」 | 调用 `patch_schedule(job_id, "paused")` |
| 「恢复监控这个群」 | 调用 `patch_schedule(job_id, "active")` |
| 「不监控了」 | 调用 `delete_schedule(job_id)`，并清理 state 文件中的记录 |
| 「看看监控了哪些群」 | 读取 state 文件，列出所有监控中的群及其状态 |
| 「修改扫描频率」 | `PATCH /schedule/{job_id}` 更新 cron |

## 填充变量说明

| 变量 | 来源 | 示例 |
|---|---|---|
| `{group_name}` | 用户配置时指定 / lark-im 查询群信息 | "团队反馈群" |
| `{chat_id}` | 用户配置时指定 / 当前群消息事件 | "oc_xxx" |
| `{window_hours}` | 用户配置时指定，默认 24 | 24 |
| `{start_time}` | 调度触发时计算：now - window_hours | "2026-07-21 22:00:00" |
| `{end_time}` | 调度触发时的 now | "2026-07-22 22:00:00" |
| `{batch_id}` | 自动生成：`batch_{YYYYMMDD}_{HHMMSS}` | "batch_20260722_220000" |
| `{members_brief}` | 读取 team.yaml 的 members 字段拼装 | 见上文格式 |

## 质量校准

定期（如每月）由用户抽查最近入库的事件，给出反馈：
- 误报率：多少候选其实不该入库 → 调整"忽略信号"规则
- 漏报率：群里明显的事件没被识别 → 调整"识别信号"敏感度
- 重复率：同事件被多次识别 → 加强去重逻辑

反馈以自然语言告诉 Agent，由 Agent 更新本模板的相关章节。

# 群消息扫描 Prompt 模板

本模板由「帮我监控这个群」生成的定时任务调用。每次触发时，Agent 拉取指定群最近一段时间的消息，按以下规则扫描并输出候选 case，最终由用户确认后入库。

## 任务上下文

- **场景**：团队管理 / 绩效记录
- **数据源**：飞书群消息（通过 `lark-im` 的 `search_messages` 或历史消息接口）
- **输出目标**：候选 case 列表，等待用户确认后写入 `~/.ethan/work/cases/{YYYY}-Q{N}/{姓名}.jsonl`
- **case 格式**：参见 `references/case-schema.md`

## 完整 Prompt

```
你是团队管理助手。下面是你的任务：扫描指定飞书群的消息，识别其中值得记录为绩效 case 的候选条目。

【群信息】
- 群名称：{group_name}
- 群 chat_id：{chat_id}
- 扫描时间窗口：最近 {window_hours} 小时（{start_time} ~ {end_time}）
- 本次扫描的批次 ID：{batch_id}（用于去重）

【团队配置】
成员清单（仅识别这些人，其他人忽略）：
{members_brief}

其中 members_brief 格式为：
- 姓名 | 角色 | 负责方向 | 关联项目

【步骤 1：拉取消息】
调用 lark-im 工具，按以下条件拉取：
- chat_id：{chat_id}
- 时间范围：start_time ~ end_time
- 排除：bot 自己发的消息、纯表情、纯「收到/好的/嗯」等无信息量回复
- 排除：本群中你已经识别并记录过的消息（通过 batch_id 或 message_id 去重）

【步骤 2：识别候选 case】
对每条候选消息，判断是否属于以下任一类别。**严格按信号判定，不要过度提取**：

A. 亮点信号（正向 case）
- 主动承担额外职责、跨团队协作
- 解决了线上问题/告警，并给出根因分析
- 提出有价值的方案/建议（被采纳或进入讨论）
- 交付物有量化成果（性能提升 X%、用户增长 Y、成本下降 Z）
- 公开分享、技术输出、文档产出

B. 质量信号（待关注 case）
- 出现 P0/P1 级别的代码/线上问题
- 同类问题重复出现（需要标注"重复"）
- 进度延期、阻塞未及时上报
- 方案有明显遗漏或风险

C. 协作信号（中性 case，可选记录）
- 主动协调资源、推动跨团队对齐
- 帮助他人解答问题、Review 给出高质量建议
- 主持/组织关键会议

**忽略信号**（以下情况不提取）：
- 日常寒暄、表情互动
- 纯进度同步（"已完成"、"已合并"）
- 无量化数据的工作汇报（除非有突出亮点）
- 来自非团队成员的消息
- 对方案/讨论的简单附和

【步骤 3：提取字段】
对每个候选 case，提取以下字段：

| 字段 | 提取规则 |
|---|---|
| `date` | 消息发送日期（YYYY-MM-DD） |
| `source` | 固定为 `group` |
| `members_brief` 姓名 | 从消息发送者或内容中识别团队成员 |
| `tags` | 根据信号类型打标签（见 case-schema.md 的 tags 推荐列表） |
| `content` | 客观事实描述，量化优先，**避免主观评价词** |
| `project` | 若消息提到项目名则填，否则留空 |
| `ref` | 飞书消息链接（lark-im 应返回 message_id 可构造链接） |
| `context` | 简要背景（如"在讨论 X 方案时"），可选 |

【步骤 4：去重与合并】
- 同一事件被多条消息讨论时，合并为一条 case，ref 字段保留主要消息链接
- 与该成员本季度已有 case 比对（读取 `~/.ethan/work/cases/{YYYY}-Q{N}/{姓名}.jsonl`）：
  - 若已有相同事件记录 → 跳过
  - 若是同类事件的后续进展 → 更新原记录或追加新记录（视情况而定）

【步骤 5：输出候选清单】
按以下格式输出，**不要直接写入 case 文件**，等待用户确认：

```
## 群扫描候选 case — {batch_id}

扫描范围：{group_name} · {start_time} ~ {end_time}
本次共识别 {N} 条候选 case：

### 候选 1
- 成员：{姓名}
- 日期：{YYYY-MM-DD}
- 标签：{tag1, tag2}
- 内容：{客观事实描述}
- 项目：{project 或 "—"}
- 引用：{消息链接}
- 背景：{context 或 "—"}

### 候选 2
...

【处理建议】
- 建议入库：候选 1, 3, 5
- 建议合并：候选 2 与已有 case {id}（同一事件）
- 建议丢弃：候选 4（信息量不足）

等待用户确认后，将"建议入库"的候选写入对应成员的 case 文件。
```

【约束】
1. **不越权写入**：未经用户确认，绝不修改 cases/ 目录下的任何文件
2. **客观描述**：case 内容只描述事实，不带主观评价（如"做得很好"）
3. **隐私边界**：识别到的 case 仅存本地，不发送到其他群或个人
4. **低噪声**：宁可漏一两件，也不要每天推 10+ 条候选让用户疲劳
5. **批次去重**：每次扫描记录 batch_id 到 `~/.ethan/work/.group_scan_state.json`，下次扫描时跳过已处理的消息
```

## 使用方式

### 用户首次配置群监控

用户：「帮我监控这个群」

Agent 行为：
1. 询问要监控的群名称或 chat_id（若消息中已 @bot 在群内则直接取当前群）
2. 询问扫描频率（默认：每天 22:00 扫描当天消息）
3. 询问时间窗口（默认：最近 24 小时）
4. 读取 `~/.ethan/work/team.yaml` 获取团队成员清单
5. 调用 `ScheduleCreateTool` 创建定时任务：
   - `category: recurring`
   - `cron: "0 22 * * *"`（或用户指定频率）
   - `prompt`: 本模板填入群信息后的完整内容
   - `title: "[群扫描] {群名称}"`
6. 在 `~/.ethan/work/.group_scan_state.json` 中记录该群的扫描状态

### 定时任务触发时

调度器按时触发，将填好的 prompt 作为新对话发给 Agent。Agent 按模板步骤执行，最终把候选清单作为回复返回。

由于定时任务的回复会发回原渠道（飞书群/私聊），用户可直接在飞书中回复「入库 1,3,5」来确认。

### 状态文件格式

`~/.ethan/work/.group_scan_state.json`:

```json
{
  "groups": {
    "{chat_id}": {
      "group_name": "团队反馈群",
      "last_scan_at": "2026-07-21T22:00:05",
      "last_batch_id": "batch_20260721_220000",
      "processed_message_ids": ["om_xxx1", "om_xxx2"],
      "cron": "0 22 * * *",
      "window_hours": 24
    }
  }
}
```

### 暂停/恢复/删除监控

| 用户指令 | Agent 行为 |
|---|---|
| 「暂停监控这个群」 | 调用 `patch_schedule(job_id, "paused")` |
| 「恢复监控这个群」 | 调用 `patch_schedule(job_id, "active")` |
| 「不监控了」 | 调用 `delete_schedule(job_id)`，并清理 state 文件中的记录 |
| 「看看监控了哪些群」 | 读取 state 文件，列出所有监控中的群及其状态 |
| 「修改扫描频率」 | `PATCH /schedule/{job_id}` 更新 cron |

## 填充变量说明

| 变量 | 来源 | 示例 |
|---|---|---|
| `{group_name}` | 用户配置时指定 / lark-im 查询群信息 | "团队反馈群" |
| `{chat_id}` | 用户配置时指定 / 当前群消息事件 | "oc_xxx" |
| `{window_hours}` | 用户配置时指定，默认 24 | 24 |
| `{start_time}` | 调度触发时计算：now - window_hours | "2026-07-21 22:00:00" |
| `{end_time}` | 调度触发时的 now | "2026-07-22 22:00:00" |
| `{batch_id}` | 自动生成：`batch_{YYYYMMDD}_{HHMMSS}` | "batch_20260722_220000" |
| `{members_brief}` | 读取 team.yaml 的 members 字段拼装 | 见上文格式 |

## 质量校准

定期（如每月）由用户抽查最近入库的 case，给出反馈：
- 误报率：多少候选其实不该入库 → 调整"忽略信号"规则
- 漏报率：群里明显的事件没被识别 → 调整"识别信号"敏感度
- 重复率：同事件被多次识别 → 加强去重逻辑

反馈以自然语言告诉 Agent，由 Agent 更新本模板的相关章节。

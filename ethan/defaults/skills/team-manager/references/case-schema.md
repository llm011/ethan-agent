# Case 记录格式规范

## 存储格式

每个成员的 case 存储为 JSONL（JSON Lines）文件，每行一条独立记录。

**文件路径**: `~/.ethan/work/cases/{YYYY}-Q{N}/{姓名}.jsonl`

## 字段定义

```json
{
  "id": "uuid-v4",
  "date": "2026-07-21",
  "source": "manual|doc|cr|group|weekly|meeting",
  "tags": ["亮点", "产出"],
  "content": "具体事实描述",
  "project": "项目名称（可选）",
  "context": "补充上下文（可选）",
  "ref": "关联链接/文档 ID（可选）"
}
```

## 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | ✅ | UUID v4，全局唯一标识 |
| `date` | string | ✅ | 事件发生日期，ISO 格式 YYYY-MM-DD |
| `source` | enum | ✅ | 来源渠道 |
| `tags` | string[] | ✅ | 分类标签，至少一个 |
| `content` | string | ✅ | 事实描述，客观、具体、可追溯 |
| `project` | string | ❌ | 关联项目 |
| `context` | string | ❌ | 补充背景，比如当时的决策环境 |
| `ref` | string | ❌ | 关联引用（MR 链接、文档 URL、消息 ID 等） |

## source 枚举值

| 值 | 含义 | 对应数据来源 |
|---|---|---|
| `manual` | 管理者手动记录 | 来源 6 |
| `doc` | 从文档中提取 | 来源 1/2 + 转发文档 |
| `cr` | 从 Code Review 中提取 | 来源 3 |
| `group` | 从群消息中提取 | 来源 4/5 |
| `weekly` | 从周报中提取 | 来源 1 |
| `meeting` | 从会议记录中提取 | 来源 2 |

## tags 推荐标签

### 正向标签
- `亮点` — 超出预期的优秀表现
- `产出` — 按时按质完成的工作交付
- `主动性` — 主动发现问题或承担额外职责
- `技术深度` — 体现技术能力的方案或实现
- `业务价值` — 对业务有明显正向影响
- `协作` — 良好的跨团队合作
- `成长` — 明显的能力提升信号
- `好建议` — 在讨论中提出有价值的思路

### 待关注标签
- `质量问题` — CR 中被指出的问题（附严重级别）
- `反馈` — 来自反馈群的问题
- `风险` — 需要关注的潜在风险
- `重复问题` — 同类问题再次出现

### 中性标签
- `需求交付` — 正常需求完成记录
- `bug修复` — 修复 bug 的记录
- `文档` — 文档产出


## 写入规则

1. **一事一记**：每条 case 只记录一件具体事实，不合并多件事
2. **客观描述**：使用事实性语言，避免「很好」「太差」等主观评价
3. **量化优先**：能用数字说明的尽量量化（时间、百分比、影响面）
4. **可追溯**：尽可能附带 ref 链接，便于绩效季回溯确认
5. **即时记录**：发生当天记录，避免事后回忆遗漏细节

# 人员日志（people/）

轻量级人员事件日志，按人按月按日记录，用于补充结构化 case（jsonl）之外的人类可读日常记录。

## 目录结构

```
~/.ethan/work/people/
├── 张三.md
├── 李四.md
├── 王五.md
└── _self.md          # 管理者自己的日志
```

- **文件命名**：`{姓名}.md`，姓名与 `team.yaml` 的 members.name 对齐
- **管理者自己**：`_self.md`
- **新建时机**：首次为某人记录时自动创建文件

## 文件格式

```markdown
# 2026-07

## 07-21
- 完成了支付模块的联调，解决了回调签名问题（14:30）
- 主动承担了新人 onboarding 培训（16:00）

## 07-20
- 在周会上提出重构库存查询方案，被采纳（10:00）

---

# 2026-06

## 06-28
- 修复了线上 P1 告警（订单超时），根因是 Redis 连接池耗尽（03:15）

---

# 2026-05

## 05-15
- 季度汇报演示，效果良好（15:00）
```

### 格式规则

| 元素 | 规则 |
|---|---|
| `# YYYY-MM` | 月份标题，顶格，按时间倒序（最新在上） |
| `## MM-DD` | 日期标题，二级标题，月份下按日期倒序 |
| `- 内容（HH:MM）` | 列表项，括号内为具体时间，24 小时制 |
| `---` | 月份之间用分隔线隔开 |
| 空行 | 月份标题前后、日期标题前后、分隔线前后各留一个空行 |

### 内容规则

1. **客观描述**：只记事实，不带主观评价（"完成了 X"、"提出了 Y"、"修复了 Z"）
2. **量化优先**：能用数字的就用数字（"性能提升 30%"、"修复 3 个 P1"）
3. **具体时间**：括号内填事件发生时间，不是记录时间；不确定时可写"（上午）"、"（下午）"、"（约 15:00）"
4. **单条单事**：一条记录只记一件事，复杂事件可拆多条
5. **去重**：同一天同一事件只记一次，后续进展另起一条

## 写入规则

### 追加到已有文件

1. 读取文件内容
2. 检查顶部是否是当前月份：
   - 是 → 在该月份下插入当天日期（若不存在），按倒序插入当天事件
   - 否 → 在文件顶部插入新月份标题 + 当天日期 + 事件，并在新月份与旧月份之间加 `---` 分隔
3. 写回文件

### 新建文件

```markdown
# {YYYY-MM}

## {MM-DD}
- {事件内容}（{HH:MM}）

```

文件末尾保留一个空行。

### 时间处理

- 默认使用当前本地时间（`Asia/Shanghai`）
- 用户指定时间时优先使用用户指定（如「记一下，张三昨天下午做了 X」→ 用昨天日期 + 下午时段）
- 不确定具体时间时写时段（上午/下午/晚上）或省略时间括号

## 使用场景

### 场景 1：用户口头记录

用户：「记一下，张三今天完成了支付模块联调」

Agent 行为：
1. 识别姓名 → `张三`
2. 识别事件 → `完成了支付模块联调`
3. 识别时间 → 今天（具体时间未指定，可询问或写时段）
4. 读取 `~/.ethan/work/people/张三.md`
   - 文件不存在 → 新建并写入
   - 文件存在 → 按写入规则追加
5. 简短回复「已记录到张三.md」

### 场景 2：群消息扫描自动记录

定时任务触发群扫描时，对识别到的候选 case，**同时**写入：
- `cases/{YYYY}-Q{N}/{姓名}.jsonl` — 结构化数据（用于绩效汇总）
- `people/{姓名}.md` — 人类可读日志（用于日常回顾）

两个文件记录同一事件，但格式不同：
- jsonl：`{"date": "2026-07-21", "tags": ["delivery"], "content": "...", "ref": "..."}`
- md：`- 完成了支付模块联调（14:30）`

### 场景 3：定期回顾

用户：「张三最近怎么样」

Agent 行为：
1. 读取 `~/.ethan/work/people/张三.md`
2. 提取最近 1-2 个月的记录
3. 按时间顺序总结关键事件
4. 结合 `cases/{YYYY}-Q{N}/张三.jsonl` 的结构化数据给出综合判断

### 场景 4：绩效撰写

用户：「汇总张三本季度的绩效」

Agent 行为：
1. 读取 `people/张三.md` 中本季度月份的记录
2. 读取 `cases/{YYYY}-Q{N}/张三.jsonl` 的结构化 case
3. 两者交叉验证，生成绩效草稿
4. people 日志提供时间线和叙事，case 提供标签和量化数据

## 与 case 的关系

| 维度 | people/*.md | cases/*.jsonl |
|---|---|---|
| 格式 | Markdown，人类可读 | JSON Lines，机器可读 |
| 粒度 | 任何事件都可记 | 仅限有绩效价值的事件 |
| 字段 | 自由文本 + 时间 | 结构化（tags/project/ref 等） |
| 用途 | 日常回顾、时间线梳理 | 绩效汇总、量化分析 |
| 写入 | 用户口头记录 + 群扫描 | 用户确认后入库 |

**互补关系**：people 日志是 case 的"草稿区"和"扩展区"。case 是 people 日志的"精华提取"。

不是所有 people 记录都会成为 case（比如「参加了个会议」可能只记 people 不进 case）；但所有 case 都应该在 people 中有对应记录（时间线追溯）。

## 隐私边界

1. **本地存储**：people/ 目录仅存本地，不发送到其他群或个人
2. **访问控制**：仅管理者本人可读写
3. **敏感信息**：薪资、绩效评级等敏感字段不写入 people 日志
4. **导出限制**：people 日志不包含在 `export_timelines` 中（仅导出 timelines.yaml + state）

# 人员日志（people/）

人员事件日志，按人按月按日记录。是 team-manager 技能的**唯一记录源**，替代之前的 cases/ 结构化存储。

## 目录结构

```
~/.ethan/work/people/
├── 张三.md
├── 李四.md
├── 王五.md
└── _self.md          # 管理者自己的日志
```

- **文件命名**：`{姓名}.md`，姓名与 `team.yaml` 的 members.name 对齐
- **管理者自己**：`_self.md`
- **新建时机**：首次为某人记录时自动创建文件

## 文件格式

```markdown
# 2026-07

## 07-21
- [亮点] 完成了支付模块的联调，性能提升 30%（14:30）
- [问题] 线上 P1 告警，Redis 连接池耗尽（03:15）
- 参加了周会（10:00）
- 和李四对齐了 Q3 OKR（15:00）

## 07-20
- 提出重构库存查询方案，被采纳（10:00）

---

# 2026-06

## 06-28
- 修复了线上 P1 告警，根因是 Redis 连接池耗尽（03:15）
```

### 格式规则

| 元素 | 规则 |
|---|---|
| `# YYYY-MM` | 月份标题，顶格，按时间倒序（最新在上） |
| `## MM-DD` | 日期标题，二级标题，月份下按日期倒序 |
| `- 内容（HH:MM）` | 列表项，括号内为具体时间，24 小时制 |
| `---` | 月份之间用分隔线隔开 |
| 空行 | 月份标题前后、日期标题前后、分隔线前后各留一个空行 |

### 标签约定

| 标签 | 用途 | 示例 |
|---|---|---|
| `[亮点]` | 突出表现、量化成果、主动承担 | `[亮点] 完成了支付联调，性能提升 30%（14:30）` |
| `[问题]` | 线上问题、质量缺陷、进度延期 | `[问题] P1 告警，Redis 连接池耗尽（03:15）` |
| `[进展]` | 已记录事件的后续更新 | `[进展] 支付联调完成（14:30）` |
| 无标签 | 普通事件（协作、会议、日常） | `参加了周会（10:00）` |

标签在内容前，用空格分隔：`- [亮点] 内容（时间）`

### 内容规则

1. **客观描述**：只记事实，不带主观评价（"完成了 X"、"提出了 Y"、"修复了 Z"）
2. **量化优先**：能用数字的就用数字（"性能提升 30%"、"修复 3 个 P1"）
3. **具体时间**：括号内填事件发生时间，不是记录时间；不确定时可写"（上午）"、"（下午）"、"（约 15:00）"
4. **单条单事**：一条记录只记一件事，复杂事件可拆多条
5. **一天多事**：同一天多个事件用多个 `-` 列表项，每条一行

## 写入规则

### 追加到已有文件

1. 读取文件内容
2. 检查顶部是否是当前月份：
   - 是 → 在该月份下查找当天日期：
     - 当天日期已存在 → 在该日期的列表末尾追加新条目
     - 当天日期不存在 → 在月份下插入当天日期标题（按倒序插入正确位置），再追加新条目
   - 否 → 在文件顶部插入新月份标题 + 当天日期 + 事件，并在新月份与旧月份之间加 `---` 分隔
3. 写回文件

### 新建文件

```markdown
# {YYYY-MM}

## {MM-DD}
- {tag}{事件内容}（{HH:MM}）

```

文件末尾保留一个空行。

### 时间处理

- 默认使用当前本地时间（`Asia/Shanghai`）
- 用户指定时间时优先使用用户指定（如「记一下，张三昨天下午做了 X」→ 用昨天日期 + 下午时段）
- 不确定具体时间时写时段（上午/下午/晚上）或省略时间括号

## 按月去重

写入前读取该成员**当月**已有记录，按以下规则处理：

| 情况 | 判断 | 处理 |
|---|---|---|
| 完全相同文本 | 精确匹配（忽略标签和时间的差异） | 静默跳过，不写入 |
| 核心动词+对象重叠 | "完成支付联调" vs "搞定了支付对接" | 询问用户：合并/替换/另记？ |
| 候选带 `[进展]` 标签 | 明确是事件更新 | 直接写入，不触发去重询问 |
| 完全不同 | — | 直接写入 |

**核心词提取**：动词（完成/修复/提出/参加）+ 对象（支付模块/告警/方案）。两者都重叠才视为疑似重复。

**跨月不去重**：跨月重复的概率很小，不做处理。如果用户发现跨月重复，手动改 md 即可。

### 去重流程示例

用户：「记一下，张三完成了支付联调」

Agent 行为：
1. 读取 `~/.ethan/work/people/张三.md` 当月记录
2. 在当月记录中查找：
   - 找到 `- 完成支付联调（14:30）` → 完全匹配 → 静默跳过，回复"已记录过"
   - 找到 `- [进展] 支付联调完成` → 核心词重叠（完成+支付联调）→ 但候选是 `[进展]` → 直接写入
   - 找到 `- 搞定了支付对接（11:00）` → 核心词重叠（搞定≈完成，支付对接≈支付联调）→ 询问"今天已有类似记录：搞定了支付对接（11:00），要合并/替换/另记？"
   - 无相关 → 直接写入
3. 用户回复后执行对应操作

## 使用场景

### 场景 1：用户口头记录

用户：「记一下，张三今天完成了支付模块联调」

Agent 行为：
1. 识别姓名 → `张三`
2. 识别事件 → `完成了支付模块联调`
3. 识别时间 → 今天（具体时间未指定，可询问或写时段）
4. 读取 `~/.ethan/work/people/张三.md`，按月去重检查
5. 写入文件
6. 简短回复「已记录到张三.md」

### 场景 2：群消息扫描自动记录

定时任务触发群扫描时，对识别到的候选事件，用户确认后直接写入 `people/{姓名}.md`。

群扫描 prompt 模板中已内置按月去重逻辑（见 `references/group-scan-prompt.md` 步骤 4）。

### 场景 3：定期回顾

用户：「张三最近怎么样」

Agent 行为：
1. 读取 `~/.ethan/work/people/张三.md`
2. 提取最近 1-2 个月的记录
3. 按时间顺序总结关键事件
4. 重点关注 `[亮点]` 和 `[问题]` 标签的事件
5. 给出综合判断

### 场景 4：绩效撰写

用户：「汇总张三本季度的绩效」

Agent 行为：
1. 读取 `people/张三.md` 中本季度月份的记录
2. 按标签分组：
   - `[亮点]` 事件 → 产出与交付、主动性与影响力
   - `[问题]` 事件 → 质量改进点
   - `[进展]` 事件 → 项目推进脉络
   - 普通事件 → 协作贡献
3. 生成绩效草稿，写入 `reviews/{YYYY}-Q{N}-draft.md`
4. people 日志提供时间线和叙事，标签提供分类维度

### 场景 5：事件检索

用户：「张三上个月修了什么 bug」

Agent 行为：
1. 读取 `people/张三.md` 中上个月的记录
2. 过滤包含"修"/"bug"/"告警"等关键词的条目
3. 按时间顺序列出

## 隐私边界

1. **本地存储**：people/ 目录仅存本地，不发送到其他群或个人
2. **访问控制**：仅管理者本人可读写
3. **敏感信息**：薪资、绩效评级等敏感字段不写入 people 日志
4. **导出限制**：people 日志不包含在 `export_timelines` 中（仅导出 timelines.yaml + state）

# 绩效汇总生成指南

## 触发时机

- 用户说「汇总绩效」「绩效报告」「绩效总结」时触发
- 绩效季开始前 2 周，Agent 主动提醒并询问是否生成

## 生成流程

### Step 1: 加载数据

1. 读取 `~/.ethan/work/team.yaml` 获取成员列表
2. 读取 `~/.ethan/work/people/{姓名}.md` 中本季度月份的记录
3. 读取 `cr-reports/` 下当季所有周报
4. 按标签统计每人事件分布（`[亮点]` / `[问题]` / `[进展]` / 普通）

### Step 2: 按人组织

为每位成员生成独立的评估板块，包含以下四个维度：

### Step 3: 输出报告

输出到 `~/.ethan/work/reviews/{YYYY}-Q{N}-draft.md`

## 评估维度

### 维度 1：产出与交付

**数据来源**: people 日志中含"完成/交付/上线"等动词的事件 + `[亮点]` 标签 + CR 周报

分析要点：
- 本季度完成了哪些需求/项目
- 代码产出量（MR 数、代码行数作为参考，不作唯一标准）
- 是否按时交付（提前/按时/延期）
- 产出的复杂度和影响面

**输出模板**:
```
#### 产出与交付

- 完成需求 N 个，关键项目：{列表}
- 代码产出：MR {N} 个，+{X} / -{Y} 行
- 交付节奏：{按时/提前/有延期}
- 代表性产出：{最重要的 1-2 个交付}
```

### 维度 2：技术质量

**数据来源**: people 日志中 `[问题]` 标签事件 + CR 周报中的质量问题

分析要点：
- P0/P1 问题数量及严重程度
- 是否存在重复出现的同类问题（技术短板）
- 正面信号：方案质量高、主动补充测试/监控
- 趋势：问题是否在减少（成长信号）

**输出模板**:
```
#### 技术质量

- CR 质量问题：P0 {N} 次，P1 {N} 次
- 重复问题：{有/无}，{描述}
- 技术亮点：{有代表性的技术方案/实现}
- 趋势：{改善/持平/恶化}
```

### 维度 3：主动性与影响力

**数据来源**: people 日志中 `[亮点]` 标签事件 + 普通事件中的协作/主动行为

分析要点：
- 主动发现/解决问题的事例
- 在群里积极回应、提出好建议
- 跨团队协作表现
- 对团队其他人的正向影响（如 review 质量高）
- 在 oncall/线上问题中的响应速度

**输出模板**:
```
#### 主动性与影响力

- 主动行为：{具体事例 1-3 个}
- 协作表现：{跨团队合作情况}
- 团队贡献：{对他人的帮助/好的 review}
```

### 维度 4：成长趋势

**数据来源**: people 日志中 `[进展]` 标签事件 + 跨时间段对比（本季度 vs 上季度，本季度前期 vs 后期）

分析要点：
- 同类问题是否减少
- 能力边界是否扩展（承担更复杂的项目）
- 技术深度是否提升
- 工作方法是否改善

**输出模板**:
```
#### 成长趋势

- 相比上季度/期初：{具体变化}
- 能力扩展：{新承担的职责/新掌握的领域}
- 待发展方向：{建议关注的成长点}
```

## 综合评估建议

在四个维度之后，给出一段综合评估建议：

```
#### 综合评估建议

**整体表现**: {一句话概括}

**核心亮点**:
1. ...
2. ...

**待改进**:
1. ...

**建议绩效区间**: {超出预期 / 符合预期偏上 / 符合预期 / 待改进}

> ⚠️ 以上仅为 Agent 基于客观数据的初步建议，最终评定请结合实际情况由管理者决策。
```

```markdown
## 团队整体观察

- 本季度团队整体产出情况
- 共性问题（如果多人有类似短板）
- 值得推广的好实践
- 团队结构/分工建议（如有）
```

---

## 注意事项

1. **不做最终判定**：建议绩效区间仅供参考，标注醒目提示
2. **数据缺失提示**：如果某人事件数过少（<3 条），明确提示「数据不足，建议补充」
3. **避免偏见**：不因事件数量多少直接等同于表现好坏（可能是记录不完整）
4. **隐私保护**：报告仅存本地，不发送到任何外部平台
5. **可编辑**：输出为 Markdown 草稿，管理者可自由修改补充

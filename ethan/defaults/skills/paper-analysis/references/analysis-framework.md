# 论文精读分析框架 — 5 维度详解

这是 Map 阶段每页产出的 JSON schema 与填写指南。**核心原则:每条结论都要能追溯到论文原文的具体数字、表格或图。看不清就回 `page_NNN.png` 再看，不许编。**

## 双路径对填写的影响(关键)

- **路径 A(vision)**:脚本把 `page_NNN.png` 喂给多模态模型,能看到图表/公式/表格的原版排版,本框架各维度都能填全。
- **路径 B(text)**:你(模型)只能读 `manifest.json` 里每页的 `text` 文字层。**图表、公式、复杂表格会失真**:
  - 架构图/流程图/定性结果图 → 文字层里**根本看不到**,`维度4_创新点.架构` 这类字段无法可靠填,需诚实标注"该页含架构图,文字层不可读,建议路径 A 重跑"。
  - 公式 → 常变乱码,如 `Attention(Q,K,V)=softmax(...)` 可能被拆成碎片,`维度4_创新点.技术原理` 要谨慎,只信能还原的。
  - 多列表格 → 文字层常把列拍平成一行,行列对应错位(见下方"常见坑"),`维度5_实验` 的数字要逐个对回文字层上下文,拿不准的标"待核"。
  - 路径 B 下,凡是"需要看图才能确定"的内容,**宁可标注不可读也不要猜**。

## 逐页 JSON schema（写入 `analysis_page_NNN.json`）

```json
{
  "page": 7,
  "维度1_摘要": {
    "核心问题": "一句话：这页/这篇要解决什么痛点",
    "解决方案": "提出的方法名 + 一句话原理",
    "关键创新": ["创新点1", "创新点2"],
    "成果": "在 <基准> 上 <指标> 从 X 提升到 Y（必须具体数字）"
  },
  "维度2_为什么重要": {
    "核心痛点": "现有方法为什么不行",
    "真实场景": "哪个实际应用必须解决它",
    "不解决的后果": "紧迫性",
    "一句话总结": "通俗解释为什么这是突破"
  },
  "维度3_前人工作": {
    "现有流派": [
      {"名称": "流派A", "代表作": "论文名(年份)", "缺陷": "带数据的具体缺陷"},
      {"名称": "流派B", "代表作": "...", "缺陷": "..."}
    ]
  },
  "维度4_创新点": [
    {"创新点": "名称", "技术原理": "通俗讲", "对比优势": "相比谁好在哪", "架构": "文字描述核心结构"}
  ],
  "维度5_实验": {
    "性能对比": [
      {"方法": "Ours", "数据集": "ImageNet", "指标": "top-1", "值": "81.5%"},
      {"方法": "SOTA-prev", "数据集": "ImageNet", "指标": "top-1", "值": "76.2%"}
    ],
    "消融": [
      {"移除模块": "模块X", "性能变化": "下降 3.2 个点", "说明该模块贡献": "..."}
    ],
    "洞察": ["从数据得出的核心结论1", "结论2"]
  },
  "本页备注": "图3是核心架构图 / 表2是主结果表 等定位信息"
}
```

> 不是每页五维全有。某页只讲实验，就只填 `维度5`，其余维度该页留空（不要写 null 占位的假内容）。**该页真正涉及的维度，必须填满且带原文数字。**

## 各维度填写要点

### 维度1 — Executive Summary
- **成果必须是数字**：✅"BLEU 从 28.4 提到 41.8" ❌"翻译质量大幅提升"
- 关键创新控制在 1-3 个，是「最核心」的，不是流水账

### 维度2 — 为什么重要
- 痛点要落到「现有方法的具体失败场景」，不是空泛地说"很重要"
- 一句话总结面向非专业读者，用类比也行

### 维度3 — 前人工作
- 至少 2 个流派，每个**点名代表论文 + 年份**
- 缺陷要带证据：✅"RNN 无法并行，训练 X 数据集需 N GPU-天" ❌"RNN 比较慢"

### 维度4 — 创新点
- 每个创新点独立成条，**与维度5 的实验一一对应**（创新点 i 由哪个实验验证）
- 技术原理用通俗语言，但不能为了通俗而失准
- 架构用文字描述清楚数据流：输入 → 模块 → 输出

### 维度5 — 实验
- **整体对比表**：列全 baseline，别只挑赢的比
- **消融**：逐个模块移除，量化每个模块的独立贡献（多少个点）
- **跨数据集/跨任务迁移**实验若有，重点分析泛化性
- 洞察 3-5 条，是「从数据里读出的判断」，不是复述数字

## Reduce 阶段整合要点

1. **合并同类项**：维度3 的流派分散在多页 → 聚成一张「前人方法对比表」
2. **数据统一**：所有维度5 的性能数字 → 一张完整主结果表 + 一张消融表
3. **因果链**：创新点(维度4) → 对应实验验证(维度5) 要能连起来，断链说明分析有漏
4. **去重**：摘要里和正文里重复的话，保留最精确的版本
5. **总体评价**：创新性/实用性/实验严谨性各打星，给出对读者研究的具体启发

## 常见坑

- **只看文字层漏图**：架构图、流程图、定性结果图只在 `png` 里，多模态看图别只读 text
- **表格串列**：PDF 文字层会把多列表格拍平成一行，数字对应关系容易错位 → 对着 `png` 核
- **公式丢失**：文字层里公式常是乱码，需要看渲染图
- **附录当正文**：appendix 的超参表/额外实验，按需处理，别和主结果混

记录提取链路从最初基线到当前的优化全过程。所有数字均来自 golden 评测集
（6 域 × 200 条）的真 LLM 评测（claude-sonnet-4.6），非离线估算。

两条评测线：
- **提取器 live**（120 条）：只测 `StructuredMemoryExtractor.extract` 本身
- **tasks 全链路**（96 条）：测 `_run_structured_extraction` 完整链路（提取 → 准入 → 落库 → job）

## 优化方法论：先解剖失败，再定向下药

每一轮都不是"改 prompt 碰运气"，而是**先把失败样本按病因分类**，再针对每类病下不同的药，最后用 golden 集验证是否命中、是否引入回退。

### 阶段一：维度注册表（R 0.43→0.74）

**诊断**：对比各域 R 发现 companion（0.88）远高于其他域，而它的 prompt 维度说明最详细。假设：**prompt 维度说明完整度 ≈ 召回率**。核对代码证实——旧 prompt 只列了约 19/64 个维度，模型根本不知道其余维度的存在（"存在性提示"缺失）。

**方法**：建维度注册表 `dimensions.py`，64 维声明式注册（每类带"角色定位"、每维带判别边界 + 正例），**校验白名单与 prompt 维度段落从同一份注册表生成**。

**结果**：R 0.56→0.74（提取器）、0.43→0.74（tasks），P 不降反升到 0.97-0.99。假设被证实：activity 0.35→1.00、decision 0.29→0.96。

### 阶段二：召回调优（R 0.74→0.81）

**诊断**：把 96 条 tasks 失败样本按 job 状态 + scenario 分类，得到三种病：
1. **零产出**（7 例，personal 重灾）——直接复现发现单句"我在腾讯工作。"模型返回空，但脱离评测环境复现却正常提取 → **随机弃权**，非系统缺陷
2. **提取遗漏**（preference fn=8）——多事实对话里模型只锚定主事实，漏掉次要明确事实；negative_pref 场景全军覆没（"别用敬语"没被识别为偏好）
3. **evidence 误判**（1-2 例）——explicit 被标 observed 压进 pending

**方法**（对病下药）：
- 病 2 → prompt 补"逐句检查每条用户消息，不要遗漏" + "别/不要/不喜欢 = 负面偏好"
- 病 1 → 空结果时复核重试一次（多轮对话追问）
- 病 3 → explicit 判据补"口语化、一句话陈述也算"

**结果**：R→0.81，personal 0.60→0.87、preference 0.50→0.62。**但引入两个回退**：负样本误提取 0→1、诊断词漏拒 0→1（"宁可多提"过头 + 复核在 companion 泄漏 case 上施压模型）。

### 阶段三：护栏微调（守住 P，R 不掉）

**诊断**：案例级明细锁定漏拒就是 `ext_comp_0026` 一例——复核重试在 companion 模式的诊断拒绝 case 上诱导了泄漏，且模型把情绪内容标成 general 维度绕过了"仅 companion 类型"的诊断词检查。

**方法**（两重防护）：
- 复核重试**只在非 companion 模式触发**（去掉施压源）
- "宁可多提候选"措辞软化为"逐句检查，不要遗漏"（去诱导性）
- 诊断词检查从"仅 companion 类型"扩到"**companion 模式下全部类型**"（确定性兜底，不依赖模型标对类型）

**结果**：四域复测 P=1.00，负样本误提取/泄漏/诊断漏拒全部回 0，companion 召回未崩（0.88），decision 反升到 0.94。

## 方法论提炼（可复用）

1. **数据驱动，不凭感觉**：每轮先跑 golden 拿分域数字，再解剖失败样本分类，最后验证。P/R/护栏三类指标同时盯，避免"提了 R 破了 P"。
2. **失败先分类再下药**：零产出 / 提取遗漏 / evidence 误判 / 边界漂移是不同的病，药方不同——统一"改 prompt"是低效的。
3. **区分随机故障与系统缺陷**：零产出能在评测里出现但脱环境复现正常 → 随机弃权，用重试兜，不必改 prompt 结构。
4. **概率提升 + 确定性兜底双管**：召回靠 prompt（概率），安全靠代码校验（确定性）；阶段二的教训是只动 prompt 会破护栏，阶段三补确定性检查才稳。
5. **护栏是硬约束**：负样本误提取/隐私泄漏/诊断漏拒任一 >0 都不接受，宁可牺牲一点召回。

## 遗留短板

- **methodology R=0.59**：根因是**维度边界漂移**——事实提取出来了，但标到相邻维度（如 execution_workflow vs analysis_reasoning），judge 严格比对 dimension 记为 fn。content 层面召回其实更高。后续方向：维度判别边界补反例、或召回评分放宽到"同类型不同维度"部分计分。
- **零产出残留**（~8/96）：部分是真负样本（正确弃权），部分是随机故障，复核重试已覆盖大部分。
- observed gate vs accrual 的 live A/B 未跑（flag 已就位，默认 gate）。
